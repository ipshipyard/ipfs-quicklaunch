**Current Challenge**: Extension uses Manifest V3 (Chrome-only) and Chrome-specific APIs. Firefox only supports Manifest V2 and uses `browser.*` APIs instead of `chrome.*`.

##### **Implementation Strategy**

**1. Dual Manifest Architecture**
- ‚è≥ Create `manifest-v2.json` for Firefox alongside existing `manifest-v3.json`
- ‚è≥ Implement build-time manifest selection based on target browser
- ‚è≥ Map Manifest V3 service worker to Manifest V2 background scripts

**2. API Normalization Layer**
- ‚è≥ Install and integrate `webextension-polyfill` package
- ‚è≥ Replace all `chrome.*` API calls with `browser.*` throughout codebase
- ‚è≥ Create browser-specific API wrappers for divergent functionality
- ‚è≥ Handle storage quota differences between browsers

**3. Background Script Compatibility**
- ‚è≥ **Chrome**: Keep existing service worker (`background.service_worker`)
- ‚è≥ **Firefox**: Convert to persistent background page (`background.scripts`)
- ‚è≥ Abstract background script logic into shared modules
- ‚è≥ Handle lifecycle differences (service worker vs. persistent page)

**4. Build System Enhancement**
- ‚è≥ Update `npm run build` to generate browser-specific distributions
- ‚è≥ Create separate output directories: `dist/chrome/` and `dist/firefox/`
- ‚è≥ Bundle webextension-polyfill only for Firefox build
- ‚è≥ Implement manifest templating system

**5. Content Script & CSP Adjustments**
- ‚è≥ Test content script compatibility with Firefox's stricter CSP
- ‚è≥ Ensure `x-ipfs-path` header detection works in Firefox
- ‚è≥ Validate DNS-over-HTTPS requests work across browsers

**6. Permission Alignment**
- ‚è≥ Audit permission differences between Chrome and Firefox

- ‚è≥ Handle browser-specific permission prompts

**7. Testing & Validation**
